# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What clSandbox is

A clean, modern, full-stack monorepo built to be **secure, fast, scalable, and cheap to host**. It is a reusable sandbox: a React front end and a Go API, containerized end to end, where **every feature ships with a visual end-to-end test**.

Priorities, in order, that drive every decision here: **security ‚Üí speed ‚Üí scalability ‚Üí hosting cost.**

## Stack (decided)

| Layer | Choice | Why |
|---|---|---|
| Frontend | React 19 + TypeScript + Vite | Fast builds/HMR, you read it easily, static output is trivial to host |
| Backend | Go (stdlib `net/http`, 1.22+ router) | Minimal deps = small attack surface; lowest latency/memory; tiny `scratch` image |
| Database | Postgres ‚Äî **Neon** (serverless) in prod, local container in dev | $0 idle, autoscale, per-PR branching; standard SQL |
| Data access | `sqlc` + `pgx` (migrations via `goose`) | Type-safe Go generated from plain SQL; injection-safe, no ORM overhead |
| Monorepo | pnpm workspaces (JS) + Go modules, orchestrated by `make` | One entrypoint over a polyglot repo |
| Containers | Docker, multi-stage; `scratch` (api) + `nginx` (web) | Smallest, safest images; cheapest to run |
| Visual E2E | Playwright (`toHaveScreenshot`) | Behavior **and** pixel baseline per feature |
| CI | GitHub Actions | api / web / e2e jobs (e2e spins up Postgres + migrates) |
| Auth | **Clerk** (managed) | JWT verified in Go (`clerkhttp`); React via `@clerk/clerk-react`. Lowest auth-bug risk; free at sandbox scale |
| Hosting | API ‚Üí **Cloud Run**, web ‚Üí **Cloudflare Pages**, DB ‚Üí **Neon** | All scale-to-zero / serverless: ~$0 idle, managed security, autoscale. See `docs/DEPLOY.md` |

**Prod data flow:** browser ‚Üí Cloudflare Pages (static React) ‚Üí `/api/*` + `/e/*` proxied via a Pages Function (generated in CI; `_redirects` can't proxy non-GET) ‚Üí Cloud Run (Go) ‚Üí Neon. Single origin, no CORS. Deploy runs on green `main` (`.github/workflows/deploy.yml`); GCP auth is keyless (Workload Identity Federation), secrets (DB URL, Clerk) via Secret Manager. Cloud Run runs `--min-instances 1 --cpu-boost` ‚Äî cold starts were the dominant perceived latency; costs a few $/mo idle (speed outranks hosting cost here; set 0 to go back to ~$0).

All six foundational decisions are made; extend the app feature by feature from here.

## This branch: the scheduler app (`app/scheduler`)

This branch builds **Whensdays** (formerly "get-togethers"), a minimal scheduling assistant, on the template. Each app in this repo is its own branch; `main` keeps the clean template plus a `gallery/` catalog (one entry + home-page screenshot per app).

**Product direction (see README "Product direction"):** the wedge is recurring small friend groups/clubs, not one-off parties or work polls. The foundational phases are **shipped** (real deploy, live branded email, guest‚Üíaccount merge, mute, host-timezone times). **North star = time-to-locked-plan**; the roadmap is organized around one loop ‚Äî create ‚Üí zero-friction invite ‚Üí visible RSVPs ‚Üí the lock moment ‚Üí recap ‚Üí re-host ‚Äî and now reads: **Now** = one-tap RSVP from email (signed links, same HMAC pattern as unsubscribe), invite-page social pressure ("4 of 7 in" + host Nudge for non-responders), a finalize celebration + shareable card, and seeding real groups (improv/stand-up/theater locals); **Next** = post-event recap thread ending in "plan the next one", group streaks, ranking poll times against ALL attendees' availability, series editing; **Later** = organizer premium (never paywall basics), deeper intent links. Live `.ics` feeds are SHIPPED: `GET /api/feed.ics?token=` (unauthenticated, HMAC `icsfeed|<uid>` token - no expiry, revoke by rotating GUEST_TOKEN_KEY; `writeVEVENT` shared with the single-event export) + authed `GET /api/calendar/feed-url` ‚Üí `FeedCard` on the Profile connections card (webcal subscribe + copy-for-Google). Comments also accept **uploaded photos**: `validGifURL` allows `data:image/`, post body cap = coverMaxBody, web `fileToPhoto` (ui.tsx, aspect-preserving 640px downscale) behind a üì∑ button. Invite card adds **WhatsApp + SMS prefilled share links** (`share-whatsapp`/`share-sms`); the Apple iCal import row covers **Outlook published links** (same URL flow, copy updated). The README's **"Viral-readiness: add / update / remove"** table is the authority on what gets built vs de-emphasized (notably: Discover/feed/follows pulled from nav until group density exists; per-type preference questions off the guest critical path; consolidate the dual availability grids). Feature breadth stays deprioritized; instrument the funnel (activation, invite‚Üíparticipant, K-factor, W4 retention) in the existing PostHog wiring before adding surface area. Watch costs: Clerk MAU (mitigated by guest flow), Klipy test-key rate limit (100/hr), calendar-data trust/scope minimalism.

## Commands

Run everything through the **Makefile** (`make help` lists targets):

- `make install` ‚Äî pnpm deps + go modules + Playwright browser
- `make dev` ‚Äî api (`go run`) + web (Vite) with hot reload
- `make build` ‚Äî web bundle + stripped static api binary
- `make test` ‚Äî Go unit tests + web typecheck
- `make e2e` ‚Äî Playwright visual tests
- `make e2e-update` ‚Äî refresh visual baselines (**review the diff before committing**)
- `make up` / `make down` ‚Äî build & run / stop the full container stack
- `make fmt` / `make lint` ‚Äî `go fmt` / `go vet`
- `make db-up` / `make db-down` ‚Äî start/stop local Postgres container
- `make migrate` / `make migrate-down` ‚Äî apply / roll back DB migrations (goose)
- `make generate` ‚Äî regenerate `internal/db` from SQL after editing `db/query` or `db/migrations`

## How the pieces fit

- **Single origin.** The browser only ever talks to the web origin; `/api/*` is reverse-proxied to the Go service ‚Äî Vite proxy in dev, nginx in the container, a CI-generated Cloudflare Pages Function in prod (`_redirects` only proxies GET). No CORS, no hardcoded API URLs.
- **Auth.** The React app wraps everything in `ClerkProvider`; protected UI sits inside `<SignedIn>`. Every API call attaches the Clerk session token (`Authorization: Bearer`). On the API, protected routes are wrapped with `clerkhttp.RequireHeaderAuthorization()`; handlers read the user id via `userIDFrom(ctx)` (the Clerk `sub`) and scope all queries to it. Never trust a user id from the request body ‚Äî always from the verified token.
- **API design.** `apps/api/main.go` connects a `pgxpool` (MaxConns raised to 16 - see below), builds `*db.Queries`, and wires routes on the stdlib mux with `securityHeaders` + `requestLogger` middleware and graceful shutdown. **Latency rule: the DB is a network hop away (Neon aws-us-west-2 - keep Cloud Run in us-west1, the same metro; it sat in us-central1 once and every query cost ~65ms), so read-heavy handlers fan independent queries out CONCURRENTLY via `parallel()` (parallel.go, stdlib WaitGroup+errors.Join)** - event detail (~14 reads), dashboard, group detail, friends, and the per-connection outbound calendar fetches all use it; N serial round trips cost N x RTT, a fan-out costs ~1. Handlers hang off `*server`, return JSON via `writeJSON`, and bound request bodies with `MaxBytesReader`. Keep dependencies minimal ‚Äî `pgx` is the only direct one.
- **Data flow (the Notes feature is the reference example).** Define schema in `db/migrations/*.sql` ‚Üí write SQL in `db/query/*.sql` ‚Üí `make generate` produces type-safe Go in `internal/db` ‚Üí call it from a handler. Never hand-write SQL strings in handlers or edit `internal/db` by hand.
- **Data caching.** `useAsync` (ui.tsx) is **stale-while-revalidate**: the last result per call site lives in a session Map (key = fetcher source + deps), so navigation renders instantly from cache while a silent refetch reconciles ‚Äî no more full-page "Loading‚Ä¶" on revisits. Every mount still refetches, so no invalidation bookkeeping; a fresh page load starts clean. `warmAsync` (ui.tsx) starts a fetch BEFORE its page mounts and parks the in-flight promise where `useAsync` finds it (requires the SAME fetcher reference ‚Äî e.g. `fetchDashboard` in lib.tsx, warmed by `ProfileGate` in parallel with the profile fetch so Home's data costs no extra round trip). **First-load path**: while clerk-js (~300KB, external) boots, `SignedIn`/`SignedOut` render nothing ‚Äî a `ClerkLoading` branch in App shows the shell + skeleton instead; `index.html` preconnects clerk + dns-prefetches the PostHog proxy. **Event page state**: series navigation reuses the mounted EventPage (only `:id` changes), so its children are keyed by event id (wholesale remount = optimistic RSVP/edit state resets) and the page-level theme-preview/celebrate state resets by hand on id change.
- **Analytics.** PostHog is wired front and back (`apps/web/src/analytics.ts`, `apps/api/internal/analytics`). Both no-op when unconfigured (dev/E2E). The web loads `posthog-js` **lazily on idle** (dynamic import; calls made before it's ready queue and flush in order) so it never weighs the critical bundle; `App` also **warms all lazy route chunks on idle** (`warmRouteChunks`) so client-side nav doesn't flash "Loading‚Ä¶". Backend owns authoritative business events + automatic `api_request` telemetry; frontend owns autocapture, pageviews, masked replay, and intent events. **Region-based consent**: EU-timezone visitors (heuristic `Europe/*`; `?consent=force` dev hook) get a `ConsentBanner` (App.tsx + `.consent-bar`) and PostHog neither loads NOR queues until Accept (`needsConsent/grantConsent/denyConsent` in analytics.ts, localStorage `whensdays.consent`); Decline is permanent for the device; US/other regions load as before, no banner (no US analytics-banner requirement; privacy page discloses). Distinct id = the app user id on both sides. **Owner alerting, two backstopping layers**: every internal 500 emails the owner via `s.alert` (alerts.go ‚Äî `s.internal` fires it, throttled 15-min PER TOPIC with suppressed-count carryover, unit-tested; recipient `ALERT_EMAIL` ‚Üí falls back to `ANALYTICS_DIGEST_TO`; no-op when email unconfigured) ‚Äî so a failed event creation or signup is never silent; total outage (app can't email) is caught by two GCP Monitoring uptime checks (whensdays.com `/` + `/api/discover`, 5-min period) with an alert policy ‚Üí the owner-email notification channel. A **daily owner digest** rides a second 8am-Pacific Cloud Scheduler job ‚Üí CRON_KEY-gated `POST /api/cron/analytics` (analytics_digest.go): two HogQL queries against PostHog's Query API (dedicated `POSTHOG_QUERY_KEY` secret scoped Query:Read ‚Äî never the ingest/personal key; `POSTHOG_APP_HOST` defaults to us.posthog.com, NOT the ingest host) ‚Üí one branded email to `ANALYTICS_DIGEST_TO` with yesterday's funnel counts + DAU + a **Free-tier runway** section (usage bars vs each provider cliff: Resend 100/day + 3k/mo via the `email_sent` PostHog event fired by `notify.OnSend`, Neon 512MB via `DatabaseSizeBytes`, PostHog 1M/mo, CF Workers 100k/day, Klipy 100/hr peak, Clerk 10k MAU ‚Äî bars go red at 80%; limits are constants in analytics_digest.go, update with pricing pages); skips silently unless key+project+recipient are all set. **See [`ANALYTICS.md`](ANALYTICS.md).**
- **Containers.** Each app has a multi-stage Dockerfile: build in a full image, ship a minimal one (`scratch` for api as non-root `65534`, `nginx:alpine` for web). This is what makes hosting cheap and the attack surface small.

## The non-negotiable workflow: a feature = code + a visual E2E test

For every feature:

1. Implement it (web and/or api).
2. **Instrument analytics** ‚Äî capture the authoritative business event(s) in the API handler (`s.analytics.Capture`) and any UI/intent event on the web (`analytics.capture`, name in `EVENTS`). Both no-op when unconfigured. See the "Instrumenting a new feature" checklist in [`ANALYTICS.md`](ANALYTICS.md).
3. Add/extend a Playwright spec in `e2e/tests/` that asserts **behavior** (`expect(...).toHaveText`, etc.) **and** a screenshot (`expect(page).toHaveScreenshot(...)`).
4. `make e2e` locally. New baselines are generated on first run ‚Äî commit the `*-snapshots/` PNGs.
5. CI re-runs the suite; a visual diff fails the build.

### Run the whole thing with only Docker (nothing else installed)

```bash
make e2e-docker
```

`compose.e2e.yaml` builds Postgres + API + web + a Playwright runner and executes the visual E2E against the real, prod-shaped stack (web nginx ‚Üí API ‚Üí Postgres). It uses **dev auth mode** so no Clerk account is needed:

- API: `AUTH_MODE=dev` swaps Clerk verification for a stub user (`demo-user`); `RUN_MIGRATIONS=true` self-applies migrations on boot. **Default is always Clerk ‚Äî dev mode is opt-in and logs a warning.**
- Web: built with `VITE_AUTH_MODE=dev` ‚Üí a Clerk-free bundle.
- E2E: `E2E_AUTH_MODE=dev` ‚Üí the spec skips Clerk sign-in.

This is the same path CI runs. Visual baselines are generated on Linux (`*-chromium-linux.png`) so they match CI exactly; commit them.

### The reference spec

`e2e/tests/notes.spec.ts` is the reference spec: it creates a note (behavior) and snapshots the stable header+form region rather than the whole page, because the notes list grows across runs. **Keep visual baselines deterministic** ‚Äî snapshot regions that don't depend on accumulated data, or reset/seed the DB in the test. Screenshots are pinned by config (`maxDiffPixelRatio: 0.01`, fixed Chromium). For an intentional UI change, run `make e2e-update`, eyeball the diff, then commit.

## Agent workflow: delegate cheap, verify smart (token economy)

For AI-assisted work in this repo, route by task type to maximize output per token:

- **Delegate to a lesser-model sub-agent** (Haiku/Sonnet) anything mechanical and well-specified: boilerplate handlers/pages that mirror an existing pattern, test scaffolding, docs sync, repetitive multi-file edits, migrations copied from a template. The delegating (stronger) model writes a tight spec first: exact files, the existing pattern to mirror (by path), and acceptance criteria.
- **Never delegate**: architecture/API-contract decisions, auth/permission/crypto code, cross-cutting refactors, anything where a subtle bug is expensive (this repo's security-first priority).
- **Verify cheaply, in this order** (stop at first failure): (1) machine gates ‚Äî `make test`, typecheck, `go vet`, targeted E2E ‚Äî these cost ~0 tokens; (2) **diff-only review** by the stronger model (`git diff` hunks, never re-reading whole files); (3) spot-check acceptance criteria only where the diff looks off.
- **One-retry rule**: if a sub-agent's output fails verification twice, the stronger model takes over directly ‚Äî retry loops cost more than doing the work.

## Conventions

- Go: tabs, stdlib-first, table-driven tests next to code as `*_test.go`. Prefer adding a dependency only when it clearly beats stdlib on security/speed.
- TS/React: strict mode on; 2-space indent; no API base URLs in components ‚Äî always relative `/api/...`.
- Secrets: `.env` (gitignored) from `.env.example`. Never commit real values.
- Decisions get recorded in this file's stack table as they're made.
- **Docs stay in sync with code.** On every change, review `README.md` and update it in the *same commit* when a user-facing feature, route, port, env var, run command, or the architecture changes. The `README.md` "Features" table is the manual-navigation guide ‚Äî keep it accurate. CI's `docs` job flags PRs that touch `apps/**` without updating `README.md`/`CLAUDE.md` (escape hatch: `[skip-docs]` in the PR title).
- **Feature screenshots are generated, never hand-edited.** After any UI change, run `make docs-shots` to recapture every feature into `docs/screenshots/` and commit the PNGs. Add a capture in `e2e/tests/screenshots.spec.ts` for each new feature/page. CI's `screenshots` job regenerates them and **warns** (non-blocking) if they differ ‚Äî the captures are full-page and render relative dates (availability grid = "today", calendar = current month), so a strict pixel gate can't stay green across days without freezing both the browser and server clocks; the fresh PNGs upload as a CI artifact for review. **The `e2e` job runs `make e2e-docker`** (hermetic dev-mode container stack ‚Äî no Clerk), so CI == local; keep it green.
- **Each containerized stack has its own compose project name** (`name:` field): `clsandbox-demo`, `clsandbox-e2e`, `clsandbox-docs`, `clsandbox` (prod). Keep them distinct so stacks don't share databases/containers.
